从零开始的LLM 4.微调与RL （代码实战篇）

发表于 2026-06-20 更新于 2026-06-23

前言

从零开始学习ai文章系列已完成《动手学深度学习》和《磨菇书》两本书的学习，新开的LLM系列课本来自《Happy-LLM》和《Hello-Agents》，但是内容和排版等个人重新进行整理，因此不会按照原来课本中的章节来写。如果哪里有错误的欢迎指正。或者不清晰的可以直接查看原文部分。

本章内容参考自 第十一章 Agentic-RL。不过原作者在讲解时使用的是一套自行封装的框架，对初学者而言抽象程度较高、不易跟随细节，因此我重新整理了一份更加"过程化"的代码实现与配套教程——尽量把每一步的输入输出、中间过程都摊开来讲，方便大家照着跑通、照着改。

在前面的章节中，我们已经系统介绍了大模型的基础知识、预训练方法和微调技术。这里先做一个小小的说明：预训练和微调在代码实现上其实高度相似，核心差异仅在于——预训练用的数据量更大、不需要套用对话模板（chat template）格式；而微调则需要将数据整理成"指令-回复"的对话形式。正因为如此，本章不再重复预训练的内容，而是聚焦于强化学习相关的两个核心方法：监督微调（Supervised Fine-Tuning，SFT） 和 群组相对策略优化（Group Relative Policy Optimization，GRPO）。

文章所有代码都能在：https://github.com/zusda/LLM-learning-from-zero/tree/main 里面找到

1. 整体规划

在正式动手之前，先梳理一下整体的实验路线，让大家对"我们要做什么、为什么这么做"有一个清晰的预期。

任务目标：以 Qwen3-0.6B 为基座模型，通过 SFT + GRPO 两阶段训练，使其具备解决 GSM8K 数学推理数据集 中小学应用题的能力，并能按照我们指定的格式输出推理过程与最终答案。

整体流程分为四大块：

  1. 数据准备

    1. 下载 GSM8K 原始数据集
    2. 分别转换成 SFT 训练格式（指令-回复对）和 RL 训练格式（仅问题，答案用于奖励计算）

  2. 模型准备 下载 Qwen3-0.6B 基座模型，封装好加载、推理、生成等常用接口函数，作为后续 SFT 和 GRPO 训练的统一基础设施。

  3. 监督微调（SFT）

    1. 编写训练所需的数据加载、Trainer、损失计算等模块
    2. 启动训练，并通过对比训练前后模型在样例问题上的表现，直观感受 SFT 带来的变化

  4. GRPO 强化学习 思路与 SFT 阶段类似：先搭建好奖励函数、采样逻辑、策略更新等核心组件，再启动训练，最后对比 SFT 后模型与 GRPO 后模型的表现差异，看看强化学习阶段到底带来了哪些提升。

2.1 GSM8K 数学推理数据集

数学推理是评估 LLM 推理能力的理想任务。首先，数学问题有明确的标准答案，可以自动评估，无需人工标注或复杂的奖励模型；其次，解决数学问题需要拆解问题、逐步推导，这正是多步推理的典型场景；最后，在数学任务上学到的推理能力具有较强的泛化性，能够迁移到其他领域。相比之下，开放式问答任务（如"如何学习编程？"）的答案质量难以客观评估，往往需要大量人工标注。

GSM8K（Grade School Math 8K）是一个高质量的小学数学应用题数据集。如下图 所示，数据集包含 7,473 个训练样本和 1,319 个测试样本，难度对应小学数学水平（2-8 年级），题型为应用题，每道题需要 2-8 步推理才能得出最终答案。

4.3 Batch 数据预处理

在上一节中，我们通过 SFTDataset 完成了单条样本的数据预处理，每次调用 __getitem__() 都会得到：

但模型训练并不是一次处理一条数据，而是一次处理一个 Batch。因此，在 DataLoader 将多条样本组合成 Batch 时，还需要进行额外处理。

最主要的问题是：

不同样本经过 Tokenizer 编码后长度通常不同，而 Tensor 要求同一维度长度一致，因此需要对 Batch 内的数据进行 Padding（补齐）。

Padding 和 Batch 整理逻辑通常定义在 DataLoader 的 collate_fn 参数中。当 DataLoader 从 Dataset 中取出多个样本准备组成一个 Batch 时，会自动调用我们传入的 collate_fn() 函数，对这些样本进行补齐、拼接等预处理操作，最终生成模型可直接训练的 Batch 数据。

处理函数如下，代码注释很清楚因此不过多解释。

需要注意区分 attention_mask 和 labels 的作用。

attention_mask 作用于 Attention 计算阶段，用于告诉模型哪些位置是真实 Token、哪些位置是 Padding。框架会根据它构造 Attention Mask，从而在构建 Attention 计算图时忽略 Padding 节点。

而 labels 作用于 Loss 计算阶段，用于告诉模型哪些位置需要参与损失计算。对于 SFT 来说，Prompt 部分和 Padding 部分通常会被设置为 -100，从而不参与 Loss 计算。

因此：

  • attention_mask 决定哪些 Token 可以参与信息交互；
  • labels 决定哪些 Token 参与 Loss 计算。

二者虽然经常同时出现，但作用于模型计算流程中的不同阶段。

不过对于 Qwen、Llama、GPT 等 Decoder-Only 模型，还需要额外说明一点：模型内部的 Causal Attention 本身就自带下三角 Mask，保证每个 Token 只能看到自己及其左侧的 Token，而无法看到未来 Token。

因此在最常见的 右侧 Padding（Right Padding）+ Decoder 模型 的训练场景下，即使将所有位置的 attention_mask 都设置为 1，右侧 Padding Token 通常也无法被前面的真实 Token 看到。在这种情况下，attention_mask 对训练结果的影响往往很小。

因为 Loss 计算阶段会直接忽略这些位置。

当然，在实际工程中仍然建议正确构造 attention_mask，以保证与 Hugging Face、Flash Attention、推理框架以及各种特殊场景保持兼容。

5.1 GRPO训练概述

在进入代码实现之前，先从整体上理解 GRPO（Group Relative Policy Optimization）的核心思想。

可以将 GRPO 理解为一种基于相对比较的强化学习训练方法：

  1. 模型针对同一个问题生成多个回答（Group）。
  2. 使用奖励函数（Reward Function）对每个回答进行评分。
  3. 在同组回答内部计算相对优劣，即优势（Advantage）。
  4. 提高高优势回答的生成概率，降低低优势回答的生成概率。
  5. 经过大量训练后，模型会逐渐倾向于生成能够获得更高奖励的回答。

从优化目标来看，GRPO 本质上并不是在向模型灌输新的知识，而是在调整模型的行为策略（Policy）。

换句话说：

SFT 更偏向于学习知识和模仿示例，而 RL 更偏向于学习“什么样的回答更值得生成”。

因此，强化学习阶段的主要作用通常是：

  • 提高回答正确率；
  • 提高推理质量；
  • 约束输出格式；
  • 减少幻觉；
  • 强化安全策略；
  • 对齐人类偏好（Human Preference）。

而不是让模型凭空学会训练数据中从未接触过的新知识。

5.1.1 GRPO 中最关键的部分是奖励来源。

通常有两种方式：

方式1：人工设计奖励函数（Rule-based Reward）

例如：

奖励函数根据各种规则进行打分：

  • 答案是否正确；
  • 格式是否符合要求；
  • 是否包含指定标签；
  • 推理过程是否完整；
  • 长度是否合理；

这类奖励函数实现简单、可解释性强，也是多数小规模实验最常见的做法。

方式2：奖励模型（Reward Model）

另一种做法是训练专门的奖励模型（Reward Model）。

5.1.2 GRPO训练面临的问题

虽然 GRPO 能够有效优化模型行为，但它并不是万能的。

问题1：探索空间不足

假设某个问题生成了 8 个回答：

此时优化方向仍然可能是错误的。

因此强化学习通常要求：

  • 基础模型本身具备一定能力；
  • 或者先经过 SFT 训练；
  • 或者增加采样数量与探索能力；

否则 RL 很容易陷入局部最优。

问题2：奖励函数决定优化方向

奖励函数本质上只是一个打分器。

它不会告诉模型：

它只会给出：

模型只能通过不断试错来寻找获得高奖励的行为。

因此：

奖励函数的质量，往往直接决定了强化学习训练效果的上限。

如果奖励函数设计存在缺陷，模型甚至可能学会“投机取巧（Reward Hacking）”，通过钻奖励规则的漏洞来获得高分，而不是真正完成任务。
